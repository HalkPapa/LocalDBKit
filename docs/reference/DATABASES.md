# データベース比較ガイド

## 1. PostgreSQL + pgvector

### 概要
- **タイプ**: リレーショナルデータベース（RDBMS）
- **特徴**: ACID準拠、強力なSQL、pgvector拡張でベクトル検索も可能

### 使用ケース
- 構造化データの保存（ユーザー、注文、在庫など）
- トランザクションが必要な処理
- 複雑なJOINクエリ
- ベクトル埋め込みの類似度検索（AI/ML）

### メリット
- データ整合性が高い
- 豊富なデータ型とインデックス
- pgvectorで1つのDBでリレーショナル + ベクトル検索
- Supabaseのバックエンドと同じ

### デメリット
- スキーマ変更が面倒
- 大規模データで遅くなることがある
- 水平スケールが難しい

### よく使う場面
- Webアプリのメインデータベース
- ECサイトの商品・注文管理
- RAG（Retrieval-Augmented Generation）システム
- ユーザー認証・権限管理

## 2. Redis

### 概要
- **タイプ**: インメモリKVS（Key-Value Store）
- **特徴**: 超高速、多様なデータ構造、Pub/Sub機能

### 使用ケース
- キャッシュ層
- セッション管理
- リアルタイムランキング・リーダーボード
- ジョブキュー（Celery, Bullなど）
- レート制限（APIスロットリング）
- Pub/Subメッセージング

### メリット
- 超高速（ミリ秒未満）
- 多様なデータ構造（String, Hash, List, Set, Sorted Set）
- シンプルで使いやすい
- Pub/Sub、トランザクション、Lua スクリプト対応

### デメリット
- メモリベースなのでデータ量に制限
- 永続化は補助的（メインストレージには不向き）
- 複雑なクエリには不向き

### よく使う場面
- API応答のキャッシュ
- ログインセッション保存
- リアルタイム通知（WebSocket + Pub/Sub）
- ページビューカウンター
- バックグラウンドジョブのキュー

---

## 3. MongoDB

### 概要
- **タイプ**: ドキュメント指向NoSQL
- **特徴**: JSONライクなドキュメント、スキーマレス、水平スケール

### 使用ケース
- 柔軟なスキーマが必要なデータ
- ネストした構造のデータ（コメント、タグなど）
- ログ・イベントデータ
- コンテンツ管理システム（CMS）
- プロトタイピング・MVP開発

### メリット
- スキーマレスで柔軟
- JSONでそのまま保存できる
- ネストしたデータを自然に表現
- 水平スケールしやすい
- 集計パイプラインが強力

### デメリット
- トランザクション機能が弱い（改善されてきている）
- JOINが苦手
- データの重複が発生しやすい
- ディスク使用量が多め

### よく使う場面
- ブログ・CMS（記事、コメント、タグ）
- IoTデータ、センサーログ
- ユーザー行動ログ
- カタログ・商品データ（ECサイト）
- リアルタイム分析

---

## 4. Qdrant

### 概要
- **タイプ**: ベクトルデータベース
- **特徴**: ベクトル類似度検索専用、フィルタリング機能、スケーラブル

### 使用ケース
- セマンティック検索
- レコメンデーションシステム
- 画像・音声の類似検索
- RAG（Retrieval-Augmented Generation）
- 異常検知

### メリット
- ベクトル検索に特化して超高速
- メタデータフィルタリングが強力
- スケーラブル
- REST/gRPC API
- 直感的なダッシュボード

### デメリット
- ベクトル検索専用（汎用DBではない）
- まだ比較的新しい技術
- 運用ノウハウが少ない

### よく使う場面
- ChatGPT風のQAシステム（RAG）
- 商品レコメンデーション
- 類似画像検索
- 文書の意味検索
- 顔認識、音声認識

## データベース選択ガイド

### どれを使うべき？

| 用途 | 推奨DB | 理由 |
|------|--------|------|
| ユーザー・商品データ | PostgreSQL | 構造化データ、トランザクション |
| キャッシュ・セッション | Redis | 超高速アクセス |
| ログ・イベント | MongoDB | スキーマレス、書き込み性能 |
| AI/ML埋め込み検索 | Qdrant または PostgreSQL+pgvector | ベクトル類似度検索 |
| リアルタイム通知 | Redis (Pub/Sub) | 低レイテンシ |
| ランキング | Redis (Sorted Set) | スコア順ソートが高速 |
| CMS・ブログ | MongoDB または PostgreSQL | 柔軟性 or 整合性 |

### 複数DB併用パターン

#### パターン1: Webアプリ標準構成
- **PostgreSQL**: メインDB（ユーザー、注文など）
- **Redis**: セッション、キャッシュ
- **Qdrant**: 商品レコメンド

#### パターン2: AIアプリ（RAG）
- **PostgreSQL**: アプリデータ、メタデータ
- **Qdrant**: ドキュメントベクトル、セマンティック検索
- **Redis**: APIキャッシュ

#### パターン3: ログ分析システム
- **MongoDB**: ログ収集、イベントストア
- **PostgreSQL**: 集計結果、レポート
- **Redis**: リアルタイムカウンター

## ベクトルDB比較: pgvector vs Qdrant

| 特徴 | PostgreSQL + pgvector | Qdrant |
|------|----------------------|---------|
| セットアップ | ✅ 簡単（拡張機能） | ⚠️ 別サービス必要 |
| パフォーマンス | ⚠️ 中規模まで | ✅ 大規模対応 |
| フィルタリング | ✅ SQLで柔軟 | ✅ 専用API |
| スケーラビリティ | ⚠️ 限定的 | ✅ 高い |
| リレーショナルデータ | ✅ 同じDBで管理 | ❌ 別途必要 |
| 学習コスト | ✅ SQL知識で可 | ⚠️ 専用API習得 |

### 推奨

- **小〜中規模、シンプル構成**: PostgreSQL + pgvector
- **大規模、パフォーマンス重視**: Qdrant
- **両方使う**: メタデータはPostgreSQL、ベクトルはQdrant

---

## まとめ

各データベースには得意分野があります：

- **PostgreSQL**: 信頼性の高い構造化データ
- **Redis**: 超高速な一時データ・キャッシュ
- **MongoDB**: 柔軟なドキュメントデータ
- **Qdrant**: AI/MLのベクトル検索

実際のアプリケーションでは、これらを組み合わせて使うのが一般的です。
